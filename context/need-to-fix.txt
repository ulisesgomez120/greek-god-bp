Plan and findings summary (scope: src services + utils, supabase functions & shared helpers, migrations reviewed where relevant).

I reviewed:

- Frontend services & utilities: src/services/auth.service.ts, database.service.ts, offline.service.ts, src/lib/supabase.ts, src/utils/storage.ts, src/utils/tokenManager.ts, plus token/storage helpers and related code.
- Supabase Edge Functions and shared helpers: supabase/functions/_shared/supabase.ts, workout-sync, sync-workouts, progression-engine, ai-progression.

Below is a structured report with: high/medium/low severity issues, redundancies, security/privacy concerns, race/consistency problems, and recommended fixes (actionable, file-level pointers). I prioritized items that can cause data loss, security risk, or produce inconsistent behavior.

High severity (should fix ASAP)

1. Duplicate, inconsistent token storage & Supabase session rehydration

   - Problem: Tokens are stored in SecureStore (src/utils/storage.ts & tokenManager) while the Supabase client (src/lib/supabase.ts) is configured to persist sessions into AsyncStorage via an AsyncStorageAdapter. That split storage means:

     - Supabase client won't automatically pick up tokens stored in SecureStore.
     - Multiple places are trying to set/rehydrate sessions (auth.service.initializeAuth, tokenManager.storeTokens → supabase.auth.setSession). This can lead to race conditions or missing sessions on cold start.

   - Files: src/lib/supabase.ts, src/utils/storage.ts, src/utils/tokenManager.ts, src/services/auth.service.ts

   - Risk: Users appear logged out; token refresh may fail or be skipped; possible leaking expired tokens; fragile auth flows.

   - Fix:

     - Consolidate token storage to a single source-of-truth. Two safe options:

       1. Use SecureStore for Supabase auth persistence by implementing and plugging a SecureStoreAdapter into createClient (preferred for security on mobile).

          - Change AsyncStorageAdapter in src/lib/supabase.ts to use SecureStore (or a wrapper that reads SecureStore and falls back to AsyncStorage).
          - Remove duplicated token writes in tokenManager/auth.service; let supabase client persist session into SecureStore.

       2. Or keep AsyncStorage for supabase auth persistence but move all token read/write logic to use AsyncStorage (not SecureStore) and encrypt tokens at rest if required.

     - Centralize token write/refresh in tokenManager only, and update auth.service to call tokenManager instead of storing tokens directly.

     - Always rehydrate Supabase via a single path: after token storage, call supabase.auth.setSession once and await it.

   - Minimal next step: pick SecureStore-backed Supabase storage or centralize to AsyncStorage; implement one adapter and update usages.

2. Conflicting/offline queue duplication and format mismatch

   - Problem: Two different offline queue implementations:

     - database.service.ts maintains offlineQueue persisted to STORAGE_KEYS.async.offlineWorkouts and has its own executeQueuedOperation logic.
     - offline.service.ts manages an enhanced queue under trainsmart_offline_enhanced_queue (prefix) with richer metadata and also calls baseAddToOfflineQueue to keep base queue in sync.

   - Files: src/services/database.service.ts, src/services/offline.service.ts, src/utils/storage.ts

   - Risk: Duplicate queues can cause missed retries, double-processing, inconsistent sync behavior, and race conditions when both try to process or delete queue items.

   - Fix:

     - Consolidate to a single offline queue implementation (prefer OfflineService enhanced queue). Make DatabaseService delegate to OfflineService for queuing rather than maintaining its own offlineQueue. Remove DatabaseService's persistence/processing or make it a thin wrapper over OfflineService.
     - Standardize queue format: choose the rich structure from OfflineService and update base helpers (utils/storage) accordingly.

   - Minimal next step: create a migration plan to convert existing stored base queue items into enhanced queue format on startup.

3. Risk of partial writes / data loss during set deletion + insert in sync functions

   - Problem: Functions delete existing sets then insert new ones without transaction semantics. If insertion fails after delete, sets are lost.

   - Files: supabase/functions/workout-sync/index.ts, supabase/functions/sync-workouts/index.ts

   - Risk: Data loss on sync failures (network / DB error).

   - Fix:

     - Use Postgres transaction semantics so delete+insert is atomic. In Supabase Edge Functions you can use a Postgres function or single upsert approach:

       - Option A: Use upsert (insert on conflict do update) for exercise_sets instead of delete then insert.
       - Option B: Wrap operations in a database-side transaction (e.g., call a stored procedure that performs replace_sets atomically).

     - If transactions are not possible in the current runtime, perform diff-based operations: upsert new/updated sets and delete only sets that are known missing, and rollback or mark sync status on failures.

4. Two overlapping sync endpoints (workout-sync vs sync-workouts) — redundancy + maintenance risk

   - Problem: There are two separate edge functions that both handle syncing but use different strategies and APIs (one uses getAuthenticatedUser helper, one handles token directly). They overlap feature-wise.

   - Files: supabase/functions/workout-sync/index.ts, supabase/functions/sync-workouts/index.ts

   - Risk: Confusion about which endpoint clients should call; difficult to keep conflict resolution consistent.

   - Fix:

     - Consolidate into one canonical sync endpoint (select the more feature-complete approach, ensure conflict resolution options cover needs).
     - Remove or deprecate the other function and update clients.

Medium severity (fix in next sprint) 5) Inconsistent naming and field mapping (snake_case vs camelCase)

- Problem: Frontend uses camelCase (weightKg, sessionId), DB uses snake_case (weight_kg, session_id). That is expected, but mapping/transform responsibilities must be consistent. DatabaseService uses transforms but other services and edge functions assume different field names (sync functions assume snake_case in some places and camelCase in others — risk of subtle mismatch).
- Files: src/services/*, supabase/functions/*
- Risk: Missing fields, incorrect comparisons, conflicts detection failures.
- Fix:
  - Ensure a central transform layer and reuse the transforms across services. Add runtime validation on payloads (e.g., zod) at sync endpoint entrypoints to normalize fields.

6. TokenManager creates its own supabase client separate from src/lib/supabase.ts

   - Problem: tokenManager constructs a new createClient instance (without storage adapter) and calls auth.refreshSession. Meanwhile other code uses default supabase import. Multiple clients may diverge in state.
   - Files: src/utils/tokenManager.ts, src/lib/supabase.ts
   - Risk: Unexpected refresh behavior; duplicate clients cause inconsistent internal caches and listeners.
   - Fix:
     - Make tokenManager use the shared supabase client (import default supabase) or configure a single client factory. If a separate client is necessary, document clearly and ensure token synchronization is robust.

7. Excessive/duplicated network status polling

   - Problem: src/lib/supabase.ts and DatabaseService both run connection checks on a 30s interval. Duplicate watchers increase network usage and complexity.
   - Files: src/lib/supabase.ts, src/services/database.service.ts
   - Fix:
     - Centralize connection monitoring in one module (lib/supabase or a dedicated connection manager) and let other services subscribe.

8. Error handling exposes internal messages and can leak info

   - Problem: Edge functions sometimes return full error messages (stack/message) to clients. E.g., AI progression returns error.message.
   - Files: supabase/functions/*
   - Risk: Information leakage (internal code, SQL errors), potential security risk.
   - Fix:
     - Return generic error messages to clients, log detailed errors internally (Sentry). Avoid echoing stack traces.

9. AI budget/limits logic inconsistency

   - Problem: ai-progression and shared helper use different budget logic. ai-progression uses hard-coded $1 budget and free tier 2 queries/month; shared helper checkAIBudget uses RPC; selectOptimalModel uses remainingBudget=1.0 in some places. The model selection and budget check aren't consistent or derived from subscription plan.
   - Files: supabase/functions/ai-progression/index.ts, supabase/functions/_shared/supabase.ts
   - Risk: Users could exceed intended spend or be blocked incorrectly; hardcoded budgets are brittle.
   - Fix:
     - Centralize budget limits in DB (subscription_plans.max_ai_queries, max_ai_cost) and use RPC check_ai_budget consistently. Ensure selectOptimalModel accepts remainingBudget param from DB/RPC, not a hard-coded 1.0.

Low severity / style / maintenance 10) Dynamic imports and require use in RN code - Problem: offline.service uses require("../../package.json") to read version; tokenManager dynamically imports "@/utils/storage" repeatedly. Dynamic imports/require add bundle complexity and affect tree-shaking. - Files: src/services/offline.service.ts, src/utils/tokenManager.ts - Fix: - Read version from ENV_CONFIG or constants at build time. Replace dynamic imports with direct imports to simplify.

11. Hash/checksum algorithm in OfflineService is naive

    - Problem: generateChecksum uses a 32-bit JS hash which may collide; better to use a robust hash (e.g., SHA-256 via Web Crypto).
    - File: src/services/offline.service.ts
    - Fix:
      - Use crypto.subtle.digest("SHA-256", ...) for stable checksums.

12. Inconsistent CORS handling across edge functions

    - Problem: Some functions use handleCors, others use corsHeaders constant. Inconsistency may lead to preflight mismatches.
    - Files: supabase/functions/*/_shared/cors.ts and functions
    - Fix:
      - Standardize on handleCors or corsHeaders helper and ensure all functions call the same preflight handler.

13. mapSupabaseError fragile pattern

    - Problem: auth.service.mapSupabaseError inspects error.message text which is brittle and locale-dependent.
    - File: src/services/auth.service.ts
    - Fix:
      - Use structured error codes (if Supabase returns them) and fallback to robust parsing; add a mapping table. Prefer error.status or error.code over message substring.

14. Use of setInterval and timers in RN: lifecycle & cleanup

    - Problem: setInterval in libraries run in app background; they may not be cleared on logout/destroy and can keep running.
    - Files: src/lib/supabase.ts, src/services/database.service.ts, tokenManager schedule timers
    - Fix:
      - Provide stop/cleanup functions; tie timers to app lifecycle events and clear them on logout.

Security & privacy notes

- Ensure all secrets (SUPABASE_SERVICE_ROLE_KEY, OPENAI_API_KEY, STRIPE keys) are only used server-side (Edge Functions) and never included in client bundles.
- Move any logging of sensitive user fields to redacted form; e.g., do not log full tokens or PII.
- Consider enabling requireAuthentication: true for SecureStore.setItemAsync where appropriate (biometric gating), depending on UX needs.
- Edge functions should sanitize/validate all incoming payloads with a schema (zod) to avoid injection attacks and malformed data.

Performance & cost

- AI Edge function uses estimated token calculation and then selects models. Ensure model selection is based on real remaining quota (from DB) and consider queuing expensive requests or returning fallback when budget is low.
- Realtime subscriptions: limit eventsPerSecond to meaningful value; use channel filters and unsubscribe carefully.
